## ⛔ Non-Negotiable Rules (apply to every task, every commit)

| Rule | Detail |
|------|--------|
| **UI never crashes** | Missing data, renamed fields, absent API keys → empty state / graceful fallback. Never a white screen or thrown error. Schema breaks are OK (TS catches them); runtime UI breaks are not. |
| **Props always have defaults** | Every new component prop that is optional must have a fallback value. Components must render with minimal data. |
| **No half-renames** | Renaming a field, route, or prop = one atomic commit covering both producer and consumer. Never leave both old and new names live at once. |
| **Reuse before creating** | Search appkit before writing a new component. Primitives → `appkit/src/ui/`. Feature components → `appkit/src/features/[domain]/`. Pages are thin wrappers only. |
| **Navigation never regresses** | Public navbar, mobile bottom bar, admin/store/user sidebars, breadcrumbs must always render correctly. Route changes require updating all link references atomically. |
| **Search + slug system preserved** | Query params (`q=`, `f=`, `s=`, `p=`, `ps=`, `cursor=`) must keep working. Slug prefixes are fixed (see Tier 0-pre). All new resource types must register their prefix. |
| **Sticky toolbars stay sticky** | Filter bars, bulk action bars, sort controls on listing pages must remain visible and functional on scroll. Never change their layout without visual verification. |
| **0 TS errors before commit** | `npx tsc --noEmit` must pass in both `letitrip.in/` and `appkit/`. Both repos committed together when both change. |
| **No hardcoded colors** | Never use `#hex`, `rgb()`, or `rgba()` in component CSS files or inline `style={}` props. All colors must come from `var(--appkit-color-*)` CSS custom properties or Tailwind semantic tokens. Only token definition files (`globals.css`, `tokens.css`) and SVG assets are exempt. Social brand colors get their own `--appkit-color-instagram` etc. tokens. |
| **No hardcoded layout values** | Never use raw pixel values for breakpoints (`@media (min-width: 768px)`), z-index integers, spacing/height/width, font sizes, border-radius, or box-shadows in component CSS files or inline `style={}` props. Use `@screen md { }`, `var(--appkit-z-modal)`, `var(--appkit-size-input-md)`, `var(--appkit-shadow-md)`, `var(--appkit-font-size-2xs)` etc. Token definition files and `tailwind.config.ts` are the only exceptions. Tailwind arbitrary syntax `p-[44px]`, `z-[50]`, `text-[10px]` is also banned — use a named token class. |

## Tier Q — Query/Sieve via Firebase Function

> **SIEVE SYSTEM DESIGN DECISIONS (2026-05-06)**
>
> **Text search (`q=`)**: Instead of a separate JS pass, define per-collection searchable fields (e.g. `title`, `description`, `tags`). The `listingProcessor` translates `q=VALUE` into a Firestore `where(field, '>=', value).where(field, '<=', value+'')` range query on each searchable field, combined with the `f=` filters in one Firestore call. This avoids the JS in-memory pass entirely. Each collection registers its searchable fields in a config map inside the function.
>
> **Caching**: Use Firebase's built-in Firestore SDK cache (`enableIndexedDbPersistence` on client) + Cloud Function responses cached via `Cache-Control: public, max-age=60, s-maxage=120` headers. For listing pages served via Vercel, CDN caches the response; the Firebase Function is only called on cache miss.
>
> **Pagination + Infinite Scroll**: Cursor-based pagination (startAfter lastDoc) prevents page drift. Function returns an opaque `cursor` token alongside items. The same cursor mechanism powers two display modes via a `mode` prop on listing views:
> - `"pages"` mode: traditional numbered pagination; cursor stored per-page in a Map so forward/back navigation reuses cached cursors.
> - `"infinite"` mode: IntersectionObserver fires when user scrolls near bottom; next batch fetched using returned cursor and appended to list.
> **Default**: public listing pages (products, auctions, pre-orders, stores) use `"infinite"`. Admin/store data tables use `"pages"`.
>
> **Filter format**: Standard sievejs expressions — `fieldName==value`, `fieldName>=value`, etc. No shorthand aliases.
